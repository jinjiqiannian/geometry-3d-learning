# 项目治理规则

## 1. Spec First

任何需求必须先输出功能规格书。

禁止直接编码。

---

### 流程

```
需求提出
  → 价值评估（High / Medium / Low）
  → 功能规格书
  → 架构评审
  → 实施计划
  → 编码
```

### 规则

- Spec 未完成，禁止进入开发
- Spec 变更，必须更新后通知所有相关方
- Spec 与代码不一致，以 Spec 为准

---

## 2. MVP First

每次只开发最小可用版本。

---

### 规则

- 禁止直接开发 V2/V3
- 先验证核心假设
- 用户验证通过后再迭代
- 禁止开发"可能有用"的功能

---

## 3. ADR First

重大架构决策必须记录。

---

### 触发条件

以下情况必须写 ADR：

- 引入新依赖
- 修改数据模型
- 修改接口协议
- 新增模块
- 技术栈变更
- 架构重构
- 性能优化方案

### 规则

- ADR 必须记录决策背景、方案对比、选择理由
- ADR 按编号归档，不可删除
- ADR 过期或失效后，用新 ADR 覆盖

---

## 4. PR Review

PR 必须有明确的范围。

---

### 检查清单

- [ ] 是否通过 CI
- [ ] 是否影响现有功能
- 是否经过自测
- [ ] 是否有对应 Spec
- [ ] 是否引入死代码
- [ ] 是否违反 ADR
- [ ] 是否有测试覆盖
- [ ] 回滚方案是否明确

### 规则

- PR 不超过 400 行
- PR 必须有清晰描述
- PR 必须关联 Issue / Spec
- 紧急修复可跳过部分检查，事后补

---

## 5. Architecture

系统架构由 SceneIR 统一管理。

---

### SceneIR 核心地位

- SceneIR 是 3D 场景的唯一真相层
- 所有渲染、交互、标注都必须通过 SceneIR
- 禁止绕过 SceneIR 直接操作 Three.js

### 模块边界

```
Input Layer     → UI / 用户交互
Processing      → AI / 解析 / 逻辑引擎
SceneIR Layer   → 3D 场景唯一真相
Presentation    → 渲染 / 动画 / 导出
```

### 规则

- 层间依赖单向：Input → Processing → SceneIR → Presentation
- 禁止 UI 直接依赖 Processing 内部
- 禁止 AI 推理与渲染逻辑混合

---

## 6. Dead Code Detection

必须持续发现并清理死代码。

---

### 检查频率

- 每次提交前自动检查
- 每周一次全量检查
- 每次重构后检查

### 死代码定义

- 未被引用的组件
- 未被调用的函数
- 被注释的代码块
- 废弃的路由
- 遗留的 API 端点
- 重复的工具函数
- 未被使用的状态

### 处理流程

1. 标记死代码
2. 评估删除风险
3. 确认无依赖后删除
4. 记录到经验沉淀

---

## 7. Lessons Learned

每次事件必须沉淀经验。

---

### 触发事件

- Bug 修复
- 线上事故
- 架构决策
- 性能优化
- 用户反馈

### 沉淀内容

- 问题描述
- 根因分析
- 修复/解决方案
- 后续预防措施

### 规则

- Lessons 写入经验沉淀文档
- 定期回顾经验库
- 经验库是团队知识资产

---

## 8. Release Process

发布必须有标准流程。

---

### 步骤

1. 代码冻结
2. 全量测试
3. 变更评审
4. 灰度发布（如有条件）
5. 全量发布
6. 发布后监控（24 小时）

### 规则

- 发布前必须确认回滚方案
- 发布窗口：避免周五晚上、节假日前
- 紧急修复可简化流程，但事后补全

---

## 9. Verification Process

每次变更必须验证。

---

### 验证层次

- 单元测试：工具函数、数据模型
- 集成测试：API、数据流
- 视觉验证：3D 场景、UI 布局
- 端到端：核心用户路径

### 规则

- 测试必须可重复
- 禁止手动测试关键路径
- 视觉变更必须截图对比

---

## 10. Rollback Process

每次发布必须有回滚能力。

---

### 回滚方案必须包含

- 回滚触发条件
- 回滚操作步骤
- 回滚后验证清单
- 回滚后数据一致性检查

### 规则

- 数据库迁移必须可逆向
- 前端发布必须可快速回退
- 回滚后必须记录根因
