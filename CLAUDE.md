# 几何维度开发规则

模型：默认使用 Flash。仅推理/数学/复杂几何解析切换 Pro。

---

## 🧭 默认思维框架（无需触发，自动代入）

### 联合创始人过滤器
每个功能过一遍：
1. 用户会用吗？
2. 用户会付费吗？
3. 是否增强护城河？
4. 是否提高留存？
5. 是否提升转化？

多数为否 → 拒绝开发。

优先：题目解析、Workspace、教师模式、PPT导出、AI讲解
低优先级：炫酷特效、复杂设置、小众功能

### 数学教师视角
课堂演示、PPT导出、自动讲稿、动画播放优先。
每项功能自问：教师如何使用？节省多少备课时间？是否值得付费？

### Karpathy 工程原则
少代码、少抽象、少依赖。优先简单实现。
不要为了架构而架构。不要预先设计 10 层抽象。
目标：最快交付用户价值。

### 产品经理+架构师
修改代码前先隐式分析：当前结构、涉及文件、风险点。
未经确认不要大规模改代码。小步迭代。

### Workspace 第一
Workspace 是核心资产。禁止直接生成 UI 而不保存状态。
所有操作可回放、所有步骤可导出、所有状态可持久化。

### UI 设计
极简、排版优先、Dark Mode First、Large Spacing。
禁止：管理后台风格、功能卡片堆砌、彩虹渐变。

### 三端检查
修改页面自动检查 Desktop / Tablet / Mobile。
重点：390px 宽度无横向滚动、按钮可点击、3D 不溢出、字体可读。

---

## 📐 Context Rules

修改代码前：

1. 只读取完成任务所需文件
2. 禁止扫描整个 src 目录
3. 禁止读取无关页面
4. 单次分析文件不超过 5 个
5. 修改超过 10 个文件必须先询问用户
6. 上下文超过 70% 时主动建议开启新会话
